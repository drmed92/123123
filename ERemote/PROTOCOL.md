# ERemote device ↔ server protocol

This is the contract between the firmware (`ERemote.ino`) and the central
server (`server/app.js`). Change either side only together with this file.

## Identity

- Device id: `d` + ESP8266 chip id in lowercase hex (e.g. `d9a3b7c`).
  Matches `^d[0-9a-f]{1,8}$`.
- Device secret: 32 lowercase hex chars, generated by the device on its very
  first boot and stored in EEPROM (survives factory reset, LittleFS format,
  and re-flashing as long as EEPROM isn't erased). The secret doubles as the
  MQTT password.
- Access code: 6 chars assigned by the server on first claim, alphabet
  `23456789ABCDEFGHJKMNPQRSTUVWXYZ` (31 chars — no 0/O/1/I/L lookalikes).
  The customer link is `<BASE_URL>/r/<code>`.

## Claim (device → server, one-time)

`POST http://<ER_HOST>/api/claim` — plain HTTP on purpose: v1 MQTT is
unencrypted, so the secret crosses the wire on every MQTT connect anyway;
TLS here would only add an ESP8266 RAM spike. First-claim-wins prevents
hijack (see below).

Request body:

```json
{ "id": "d9a3b7c", "secret": "<32 hex>", "fw": "1.0" }
```

Responses:

- `200` `{ "ok": true, "code": "K7M3Q9", "link": "https://host/r/K7M3Q9",
  "mqtt": { "host": "host-or-ip", "port": 1883 } }` — new device registered,
  or an already-known device re-claiming with its correct secret (idempotent;
  same code returned every time).
- `403` `{ "ok": false, "error": "secret-mismatch" }` — id exists with a
  different secret. The rightful device keeps its registration; a fresh
  device flashed with a reused chip id cannot take over. Admin can delete
  the stale record (see server/deploy.md) to allow re-registration.
- `400` — malformed id/secret.

The device retries the claim every 60 s until it gets a 200, then never
claims again (identity stored in EEPROM).

## MQTT (device ↔ broker on the server, port 1883)

- Username: device id. Password: device secret. The broker's authorizer
  restricts each device to its own topics: `er/<username>/#`.
- `er/<id>/cmd` — server → device. Payload `{"btn":"on"}` (or bare
  `on`/`off`/`eco`). Device validates the button and that an IR code is
  recorded, then transmits immediately.
- `er/<id>/state` — device → server, **retained**. Published on connect, on
  any change (code recorded, genset appear/disappear, command executed) and
  at least every 60 s:

```json
{ "online": true,
  "codes": { "on": true, "off": false, "eco": false },
  "genset": false,
  "rssi": -61 }
```

- LWT: the device registers a retained will `{"online":false}` on its state
  topic, so the server (and web page) see the device drop within the
  keepalive window even on power loss.

## Web page (browser ↔ server, HTTPS via Caddy)

- `GET /r/<code>` — the control page (404 for unknown codes; heavily
  rate-limited per IP to make 6-char code guessing impractical).
- `GET /api/r/<code>/state` — latest cached state + `lastSeen` (ms since
  epoch) or `online:false` if never seen.
- `POST /api/r/<code>/cmd` body `{"btn":"on"}` — publish to the device.
- `GET /api/r/<code>/events` — Server-Sent Events stream; each event is the
  same JSON as `/state`, pushed whenever the device publishes.
